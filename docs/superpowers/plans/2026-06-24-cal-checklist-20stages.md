## Locked design decisions (from two adversarial-review rounds)
- **Checklist** (high-confidence): extract out of content.js; smart views Today/Upcoming/Overdue; p1–p4 priorities. **Priority is UI-only — it NEVER bypasses `requires[]` dependency locks.** `checklistItems()` stays an importable pure export (dashboard.js + readiness depend on it). p1–p4 needs a `jwh-check-priority-v2` map + migration (v1 array→p1).
- **Calendar week view** (hardened): **drag-reschedule on SINGLE-DAY events ONLY** (multi-day seasonal bars are click-to-edit, never draggable — dragging a 5-month bar would shift the whole window). Use the app's **real `--c-*` category colors** (festival #c43459, music #5a3fb5, seasonal #8a6508, … all defined in style.css:33-45 with `-ink` dark variants). **Today = red outline** (`box-shadow: inset 0 0 0 1px var(--red)`), NOT a filled circle. **Lane-pack** multi-day bars (greedy), don't give each its own full-width row. Per-day always-visible **"+" add** affordance. Mobile = **vertical 7-day list** (reuse agenda rendering bounded to the week; no horizontal grid → no route-swipe conflict).
- **CUT** (do not build): hourly time-grid, timed events, sub-tasks, recurring, NL quick-add.
- Conventions: zero-build ES modules; `esc()` every dynamic string (escapes `& < > "`, double-quote attrs); single-dispatch `jwh:data-changed` (no render() at mutation site, render() never dispatches); versioned KEYS; bump SW `CACHE` + precache new `assets/*.js`; pure libs unit-tested (node --test).
